## 1. Background

SILO currently has a working tableau simplex MVP for a narrow standard form:

```text
maximize   c^T x + c0
subject to A x <= b
           x >= 0
           b >= 0
           x continuous
```

The current MVP already supports:

- slack-variable construction for `<=` rows;
- deterministic entering-column rule;
- deterministic leaving-row ratio test;
- pivot operation;
- optimal status;
- unbounded status;
- unsupported-model rejection;
- primal values for original variables;
- objective value including objective constant;
- tests for standard-form LPs and unsupported inputs.

The next step is to add a readable educational **Phase I / Phase II tableau flow** so that the solver can handle:

```text
<= rows
>= rows
= rows
negative RHS rows after normalization
infeasible LP detection
```

This task is still part of the educational dense tableau solver. It is not meant to become a production-grade LP solver.

------

## 2. Goal

Extend the tableau simplex implementation from the current standard-form MVP to a small but complete Phase I / Phase II tableau solver for continuous LPs with linear rows of type `<=`, `>=`, and `=`.

The solver should be able to:

1. normalize rows with negative RHS;
2. add slack, surplus, and artificial variables where needed;
3. run Phase I to find an initial feasible basis;
4. detect infeasibility when the Phase I artificial objective cannot be driven to zero;
5. remove or ignore artificial variables before Phase II;
6. solve the original objective in Phase II;
7. preserve existing MVP behavior and tests;
8. keep the implementation readable and deterministic.

## 6. Mathematical Scope

### 6.1 Accepted model class

This task should support continuous LPs with:

```text
maximize   c^T x + c0
subject to linear rows with senses <=, >=, =
           variables x >= 0
           no finite variable upper bounds
           no nonzero variable lower bounds
           continuous variables only
```

### 6.2 Row normalization

Before constructing the tableau, normalize each row so that RHS is nonnegative.

For a row:

```text
a_i x <= b_i
```

if `b_i < 0`, multiply by `-1` and flip the sense:

```text
-a_i x >= -b_i
```

For:

```text
a_i x >= b_i
```

if `b_i < 0`, multiply by `-1` and flip the sense:

```text
-a_i x <= -b_i
```

For:

```text
a_i x = b_i
```

if `b_i < 0`, multiply by `-1`; equality remains equality.

Use the existing numerical tolerance convention when deciding whether a RHS is effectively zero.

### 6.3 Slack, surplus, and artificial variables

For each normalized row:

#### `<=` row

```text
a_i x + s_i = b_i
s_i >= 0
```

Add a slack variable. The slack variable can enter the initial basis.

No artificial variable is needed.

#### `>=` row

```text
a_i x - r_i + a_i_art = b_i
r_i >= 0
a_i_art >= 0
```

Add a surplus variable and an artificial variable.

The artificial variable enters the initial basis.

#### `=` row

```text
a_i x + a_i_art = b_i
a_i_art >= 0
```

Add an artificial variable.

The artificial variable enters the initial basis.

## 7. Phase I Requirements

Implement Phase I in a readable way.

The Phase I objective should minimize the sum of artificial variables:

```text
min sum artificial variables
```

For the existing maximization tableau framework, this can be represented as maximizing:

```text
- sum artificial variables
```

or by another clear convention. The convention must be documented in code comments and tests.

Phase I should:

1. construct an initial tableau with a feasible artificial basis;
2. optimize the Phase I objective;
3. conclude infeasible if the artificial objective is nonzero beyond tolerance;
4. conclude feasible if the artificial objective is zero within tolerance;
5. prepare the tableau for Phase II.

Important: the Phase I objective row must be canonical with respect to the initial artificial basis. If artificial variables are initially basic, the objective row must be adjusted so the basic artificial columns have zero coefficients.

------

## 8. Transition from Phase I to Phase II

After Phase I succeeds:

1. artificial variables should not be allowed to enter during Phase II;
2. artificial-variable columns may be removed if this is simpler and safe;
3. if artificial variables are kept, pricing must ignore them in Phase II;
4. the original objective row must be restored and made canonical with respect to the current basis;
5. the original objective constant must be preserved.

The preferred approach is:

```text
Phase I tableau
  -> prove feasibility
  -> remove artificial columns
  -> rebuild original objective row over remaining columns
  -> canonicalize objective row using current basis rows
  -> run Phase II
```

If removing artificial columns is complex, keep them but ensure they are never selected as entering variables in Phase II. In that case, document the reason clearly.

## 10. Status and Messages

Use existing statuses:

```text
OPTIMAL
INFEASIBLE
UNBOUNDED
ITERATION_LIMIT
ERROR
```

Expected behavior:

- Unsupported variable type, finite upper bound, or nonzero lower bound -> `ERROR`
- Phase I artificial objective nonzero -> `INFEASIBLE`
- Phase II improving column with no leaving row -> `UNBOUNDED`
- Phase II optimum found -> `OPTIMAL`
- Iteration limit reached in either phase -> `ITERATION_LIMIT`

Messages should be clear enough to show which phase failed when relevant, for example:

```text
Phase I detected infeasibility.
Phase II detected an unbounded improving direction.
Tableau simplex solved the LP.
```

## 11. Pricing and Ratio Test

Keep deterministic behavior.

Entering column:

- choose the first eligible negative objective-row coefficient below tolerance;
- ignore RHS column;
- ignore artificial columns during Phase II if artificial columns are kept.

Leaving row:

- use the minimum nonnegative ratio among rows with positive pivot coefficient;
- break ties by row order;
- use existing tolerance.

Do not introduce advanced pricing rules.

           x <= 5

```

```

### 12.3 Equality row requiring artificial variable

Test a feasible LP with an equality row.

Example:

```text
maximize x + y
subject to x + y = 4
           x <= 3
           y <= 3
           x, y >= 0
```

Expected:

```text
status = OPTIMAL
objective = 4
x and y satisfy x + y = 4
x <= 3
y <= 3
```

Do not require a unique solution if the optimum is degenerate or has multiple optima. Check feasibility and objective value.

### 12.4 Negative RHS normalization

Test a row with negative RHS.

Example:

```text
maximize x
subject to -x <= -2
           x <= 5
           x >= 0
```

This is equivalent to:

```text
x >= 2
x <= 5
```

Expected:

```text
status = OPTIMAL
x = 5
objective = 5
```

### 12.5 Infeasible LP

Test an infeasible LP requiring Phase I detection.

Example:

```text
maximize x
subject to x >= 2
           x <= 1
           x >= 0
```

Expected:

```text
status = INFEASIBLE
message mentions Phase I or infeasibility
```

### 12.6 Feasible equality with zero RHS

Test an equality row with zero RHS if simple.

Example:

```text
maximize y
subject to x = 0
           y <= 3
           x, y >= 0
```

Expected:

```text
status = OPTIMAL
y = 3
x = 0
objective = 3
```

### 12.7 Unsupported model tests remain

Unsupported tests should remain for:

```text
finite variable upper bounds
nonzero lower bounds
integer variables
binary variables
```

After this task, `>=` and `=` constraints should no longer be unsupported.

Update old tests that expected `>=` or `=` to return `ERROR`.
