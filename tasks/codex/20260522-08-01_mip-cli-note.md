## 1. Background

SILO now has:

- LP JSON examples under `examples/json/`;
- MIP JSON examples under `examples/mip/`;
- tableau simplex LP backend;
- revised simplex LP backend;
- branch-and-bound Python API for binary and bounded nonnegative integer MIPs;
- `silo solve MODEL_PATH` for LP solving;
- `silo solve MODEL_PATH --solver tableau`;
- `silo solve MODEL_PATH --solver revised`;
- `silo compare MODEL_PATH` for LP backend comparison;
- `silo presolve MODEL_PATH` for presolve diagnostics;
- `silo solve MODEL_PATH --presolve` for explicit LP solve-time presolve.

MIP solving currently exists through the Python API:

```python
from silo.io.json_reader import read_json_model
from silo.mip.branch_and_bound import BranchAndBoundSolver

model = read_json_model("examples/mip/binary_knapsack.json")
solution = BranchAndBoundSolver().solve(model)
```

There is no public MIP CLI yet.

The previous task intentionally added MIP JSON examples and regression tests before exposing a CLI. This task should decide how MIP solving should be exposed through CLI in a way that does not confuse it with the existing LP-oriented `silo solve`.

This is a design-only task.

## 7. Content Requirements

### 7.1 Purpose

Explain that MIP solving is now available through the Python API, but not CLI.

The design should protect the existing LP-oriented CLI semantics:

```bash
silo solve MODEL_PATH
silo solve MODEL_PATH --solver tableau
silo solve MODEL_PATH --solver revised
```

The goal is to expose MIP solving without making `silo solve` ambiguous.

------

### 7.2 Current CLI Boundary

Summarize current commands:

```text
silo solve       -> LP solve using tableau/revised backend
silo compare     -> compare LP tableau/revised backends
silo presolve    -> presolve/scaling diagnostics
```

Clarify that current `--solver` means LP backend, not MIP algorithm.

Current LP solver choices:

```text
tableau
revised
```

MIP branch-and-bound uses an LP backend internally, but it is not itself an LP backend.

------

### 7.3 Why MIP Needs an Explicit CLI Boundary

Explain the risk of doing this too casually:

```bash
silo solve examples/mip/binary_knapsack.json
```

could be ambiguous because `solve` currently routes to LP backends.

A MIP model contains binary/integer variables that LP backends reject, while `BranchAndBoundSolver` can solve them.

Therefore, the CLI should make the modeling class explicit:

```text
LP solve and MIP solve are different command paths.
```

------

### 7.4 Candidate Command Designs

Compare at least these options.

#### Option A: `silo mip-solve MODEL_PATH`

Example:

```bash
silo mip-solve examples/mip/binary_knapsack.json
```

Pros:

```text
clear separation from LP solve
simple to document
does not overload --solver
easy to keep LP and MIP semantics distinct
```

Cons:

```text
adds a new top-level command
slightly less elegant than a unified solve command
```

#### Option B: `silo solve MODEL_PATH --mip`

Example:

```bash
silo solve examples/mip/binary_knapsack.json --mip
```

Pros:

```text
keeps one solve command
explicit flag tells CLI to use MIP path
```

Cons:

```text
solve becomes more complex
users may forget --mip and get LP errors
future interactions with --solver and --presolve become confusing
```

#### Option C: `silo solve MODEL_PATH --solver branch-and-bound`

Example:

```bash
silo solve examples/mip/binary_knapsack.json --solver branch-and-bound
```

Pros:

```text
uses existing --solver concept
```

Cons:

```text
confuses LP backend with MIP algorithm
currently --solver means tableau/revised LP backend
branch-and-bound itself still needs an LP backend
```

This option should likely be rejected for the first version.

#### Option D: `silo mip MODEL_PATH` or `silo mip solve MODEL_PATH`

Example:

```bash
silo mip examples/mip/binary_knapsack.json
```

or:

```bash
silo mip solve examples/mip/binary_knapsack.json
```

Pros:

```text
could scale to MIP subcommands later
```

Cons:

```text
current argparse structure is simple and not subcommand-oriented
may be too much CLI structure too early
```

------

### 7.5 Recommended First Command

Make a recommendation.

Preferred recommendation:

```bash
silo mip-solve MODEL_PATH
```

Reason:

```text
it is explicit;
it avoids changing existing silo solve behavior;
it keeps LP backend selection separate from MIP algorithm selection;
it fits the current simple argparse command style;
it is easy to test.
```

The first command should be:

```bash
silo mip-solve examples/mip/binary_knapsack.json
```

and optionally:

```bash
silo mip-solve examples/mip/binary_knapsack.json --output outputs/knapsack_solution.json
```

------

### 7.6 First-Version Flags

Recommend first-version flags:

```text
--output / -o
--lp-solver tableau|revised
--node-limit N
```

#### `--lp-solver`

MIP branch-and-bound internally solves LP relaxations.

Recommended:

```bash
silo mip-solve examples/mip/binary_knapsack.json --lp-solver tableau
silo mip-solve examples/mip/binary_knapsack.json --lp-solver revised
```

Default:

```text
tableau
```

Do not reuse `--solver` for this first MIP command because `--solver` already means LP backend for `silo solve`.

#### `--node-limit`

Expose:

```bash
silo mip-solve examples/mip/binary_knapsack.json --node-limit 1000
```

Default should match `BranchAndBoundSolver`.

Validate nonnegative integer.

#### `--output`

Use the same output behavior as `silo solve`.

------

### 7.7 Output JSON and Exit Codes

Recommend using the existing `Solution` JSON schema:

```text
status
objective_value
primal_values
slack_values
dual_values
reduced_costs
basis_status
message
```

For MIP output:

```text
status
objective_value
primal_values
message
```

will be populated.

These may remain empty:

```text
slack_values
dual_values
reduced_costs
basis_status
```

Do not add MIP node counts to the first public solution JSON unless a separate `--details` flag is designed.

Exit codes should mirror `silo solve`:

```text
0 = status optimal
1 = non-optimal status or read/write error
2 = argparse usage error
```

------

### 7.8 Relationship to Presolve and LP Backends

Clarify:

```text
MIP presolve is not implemented.
LP presolve is separate.
```

First MIP CLI should not support:

```text
--presolve
```

unless a separate design task defines how MIP presolve should behave.

The first MIP CLI may support LP backend selection through:

```text
--lp-solver
```

because branch-and-bound internally solves LP relaxations.

This does not mean tableau/revised are MIP solvers. They are LP backends.

The future CLI documentation should reference:

Define tests for the future implementation.

Required future tests:

```text

mip-solve integer_allocation -> optimal 7
```

