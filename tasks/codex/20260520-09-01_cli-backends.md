## 1. Background

SILO currently has two native LP solver implementations:

1. dense tableau simplex with Phase I / Phase II;
2. revised simplex with Phase I / Phase II.

The CLI currently supports solving JSON models through the existing default solve path:

```bash
silo solve MODEL_PATH
silo solve MODEL_PATH --output OUTPUT_PATH
```

The next step is to allow users to explicitly choose which native solver backend is used from the command line.

This task should add:

```bash
silo solve MODEL_PATH --solver tableau
silo solve MODEL_PATH --solver revised
```

The default backend should remain `tableau` unless there is already a documented default. This avoids changing existing CLI behavior unexpectedly.

------

## 2. Goal

Add a small, explicit CLI solver-backend selection mechanism.

After this task, these commands should work:

```bash
silo solve examples/json/production.json
silo solve examples/json/production.json --solver tableau
silo solve examples/json/production.json --solver revised
silo solve examples/json/production.json --solver revised --output outputs/production_revised_solution.json
python -m silo.cli.main solve examples/json/production.json --solver revised
```

The CLI should:

1. read the JSON model;
2. select the requested native solver;
3. solve the model;
4. output deterministic solution JSON to stdout or file;
5. preserve existing exit-code conventions.

## 6. CLI Interface Requirements

Update the `solve` command to support:

```text
--solver
```

Allowed values:

```text
tableau
revised
```

Default:

```text
tableau
```

Recommended argparse behavior:

```bash
silo solve MODEL_PATH --solver tableau
silo solve MODEL_PATH --solver revised
```

Invalid solver names should be handled by argparse and return usage-error exit code `2`.

Do not accept arbitrary solver strings.

Do not implement `--solver highs`, `--solver glpk`, or external solver backends.

## 7. Solver Selection Requirements

Add a clear internal mapping from solver name to solver class.

Example:

```python
SOLVER_FACTORIES = {
    "tableau": TableauSimplexSolver,
    "revised": RevisedSimplexSolver,
}
```

or a small helper function:

```python
def create_solver(name: str) -> LPSolver:
    ...
```

The helper should be simple and testable.

The CLI should not silently fall back from revised to tableau or tableau to revised.

If the selected solver returns `ERROR`, `INFEASIBLE`, `UNBOUNDED`, or `ITERATION_LIMIT`, the CLI should output that solver's solution JSON and return exit code `1`.

If the selected solver returns `OPTIMAL`, return exit code `0`.

## 8. Exit Code Requirements

Preserve the existing exit-code convention:

```text
0 = selected solver returned OPTIMAL
1 = solve command ran but selected solver returned non-optimal or error status
2 = argparse usage error
```

This task should not introduce additional exit-code categories.

## 9. Solution JSON Requirements

The CLI should continue to output deterministic solution JSON.

The output schema should remain:

```json
{
  "basis_status": {},
  "dual_values": {},
  "message": "...",
  "objective_value": 21.0,
  "primal_values": {},
  "reduced_costs": {},
  "slack_values": {},
  "status": "optimal"
}
```

If the selected solver is revised, the output should be the revised solver's `Solution`.

If the selected solver is tableau, the output should be the tableau solver's `Solution`.

Do not add a new top-level `"solver"` field in the solution JSON unless already present elsewhere. If you think solver name should be recorded, mention it in the execution report as a possible future enhancement rather than adding it now.
