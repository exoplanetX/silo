## Required README Additions

Add a new section to `tasks/README.md` defining the SILO Development Operating System.

The new section must cover at least the following rules.

### 1. SILO-DOS Purpose

Define SILO-DOS as the operating protocol that governs how solver development knowledge is converted into controlled Codex tasks.

It should clarify that:

- phase files contain long-term development knowledge;
- Codex task files contain immutable issued task contracts;
- report files contain execution memory;
- Codex must not treat `tasks/` as a general notes folder.

### 2. Phase-to-Task Rule

Define how `tasks/phases/` files are converted into `tasks/codex/` task files.

The rule should state that:

- a phase is not itself a task;
- a phase contains goals, scope, expected files, tests, acceptance criteria, and development knowledge;
- a Codex task must be traceable to a phase item, explicit maintenance need, or user-approved process task;
- only one atomic task should be issued at a time.

### 3. Atomic Task Rule

Define the atomic task contract.

Every Codex task should include:

- task metadata;
- objective;
- phase reference;
- task type;
- scope lock;
- allowed changes;
- forbidden changes;
- required tests or checks;
- acceptance criteria;
- stop conditions;
- expected report path;
- Git mode.

The rule should state that one task must solve exactly one primary problem.

### 4. Scope Lock Rule

Define that Codex must stay within the allowed files and allowed objective.

The rule should state that if Codex discovers a broader issue, it should document the issue in the report instead of fixing it within the same task.

### 5. Git Mode Rule

Add the following Git modes:

```text
no-git
local-commit
push-on-success
sync-only
```

Define them as follows:

- `no-git`: modify files only; do not commit or push.
- `local-commit`: commit locally after successful checks; do not push. This is the default mode.
- `push-on-success`: commit locally and attempt one push only after successful checks.
- `sync-only`: do not modify project files; only synchronize already committed local changes with the remote repository.

The rule should also state that push failure is non-fatal. If push fails, Codex must preserve the local commit and document the failure in the report.

### 6. Execution Report Rule

Define that every executed task must create a report under `tasks/reports/` using the matching task ID and slug.

The report should include:

- task objective;
- files changed;
- checks or tests run;
- results;
- deviations from scope, if any;
- Git status before and after;
- local commit hash, if created;
- push attempted or skipped;
- unresolved issues;
- next recommended atomic task.

### 7. One-Step Execution Rule

Define that Codex must stop after one atomic task.

Codex must not automatically continue to the next task unless the user explicitly asks it to continue.

### 8. ChatGPT / Codex Responsibility Boundary

Define the intended responsibility split:

- ChatGPT or the user may design phase strategy, revise high-level direction, and approve phase transitions.
- Codex executes one task at a time according to task contracts and local repository rules.
- Codex should not depend on a live ChatGPT conversation to interpret task boundaries.
- Codex should rely on `tasks/README.md`, phase files, issued task files, and prior reports.
