## 1. Background

SILO currently has two native LP solver paths:

1. dense tableau simplex with Phase I / Phase II;
2. revised simplex with Phase I / Phase II.

The tableau solver is the older educational reference implementation. The revised simplex solver is the newer basis-oriented implementation.

The revised simplex solver now supports small continuous maximization LPs with:

```text
<=, >=, and = rows
negative RHS normalization
nonnegative continuous variables
no finite upper bounds
no nonzero lower bounds
maximization only
```

Both solvers return public `Solution` objects with fields such as:

```text
status
objective_value
primal_values
slack_values
dual_values
reduced_costs
basis_status
message
```

However, before moving to warm starts, CLI backend selection, or MIP, the public diagnostics of the revised simplex solver should be checked against the tableau solver on small deterministic LPs.

This task should also add a short design note for future dual-value reporting, but it should **not** implement dual values yet.

## 6. Diagnostic Alignment Scope

Compare these public fields between tableau and revised simplex where meaningful:

```text
status
objective_value
primal_values
slack_values
reduced_costs
basis_status
dual_values
```

### 6.1 Must match

For nondegenerate or deterministic cases, these should match:

```text
status
objective_value
primal_values
slack_values
```

### 6.2 Reduced costs

Reduced costs should use a consistent public convention across both solvers.

For maximization with variables at lower bound zero, the public convention should be:

```text
reduced_cost_j = c_j - pi^T A_j
```

At optimum:

```text
nonbasic variables at lower bound should have reduced_cost <= tolerance
basic variables should have reduced_cost approximately 0
```

If the tableau internal row stores the opposite sign, convert before exposing public `Solution.reduced_costs`.

Add tests that fail if tableau and revised simplex expose opposite reduced-cost signs.

### 6.3 Basis status

Basis status may differ in degenerate cases or multiple-optimum cases.

Only require exact basis-status parity on examples where the basis is deterministic.

Otherwise, test that each solver returns valid statuses:

```text
basic
nonbasic_lower
```

for all original variables.

### 6.4 Dual values

Both solvers should continue to leave:

```python
dual_values == {}
```

Do not add public dual values in this task.

```markdown
## 5. Equality, <=, and >= Row Mapping

## 6. Testing Requirements Before Exposure

## 7. Out of Scope for Now
```

### 10.1 Purpose

Explain why `dual_values` should not be exposed casually.

### 10.2 Current status

State that `dual_values` currently remains empty for tableau and revised simplex.

### 10.3 Candidate convention

For standard form:

```text
maximize c^T x
subject to A x = b
           x >= 0
```

the revised simplex multiplier is:

```text
pi^T = c_B^T B^{-1}
```

This is the natural candidate for equality-form row duals.

### 10.4 Row normalization issues

Explain that user-facing constraints may be:

```text
<=
>=
=
```

and rows may be multiplied by `-1` during normalization. Therefore, raw standard-form multipliers must be mapped carefully back to original constraints.

### 10.5 Testing requirements

Before exposing `dual_values`, tests should check:

```text
shadow-price sign for <= rows
shadow-price sign for >= rows
equality row multipliers
negative RHS row normalization
comparison with reduced costs
complementary slackness on small LPs
```
