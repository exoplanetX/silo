## 1. Background

SILO currently has a working educational dense tableau simplex solver with:

- Phase I / Phase II support;
- `<=`, `>=`, and `=` row support;
- infeasible and unbounded status detection;
- JSON model input;
- CLI solve workflow;
- solution JSON output with primal values, slack values, reduced costs, and basis status;
- user-facing documentation and JSON examples.

The next major solver layer is Phase 3:

```text
Revised Simplex and Basis Reoptimization
```

## 7. Content Requirements

### 7.1 Purpose

Explain that tableau simplex is useful for learning but not suitable as the long-term LP engine.

Revised simplex should become the bridge toward:

```text
basis-oriented LP solving
warm starts
MIP node reoptimization
future decomposition methods
```

Do not claim industrial performance.

### 7.2 Current Tableau Solver Boundary

Summarize the current tableau solver’s role:

```text
educational dense tableau
Phase I / Phase II
small continuous LPs
CLI solve workflow
```

Explain that it should remain as a reference implementation and test oracle for small instances.

### 7.3 Mathematical Form

Specify the first revised simplex target form.

Recommended first target:

```text
maximize c^T x + c0
subject to A x = b
           x >= 0
```

with a transformation layer from SILO’s current row-sense form:

```text
<=, >=, =
```

to equality form using slack/surplus/artificial variables.

Mention that finite upper bounds and nonzero lower bounds remain future work.

### 7.4 Basis Representation

Specify a future `Basis` abstraction.

It should record at least:

```text
basic column indices
nonbasic column indices
variable names
basis status for variables
```

Potential future status labels:

```text
basic
nonbasic_lower
nonbasic_upper
fixed
```

For the first implementation, `basic` and `nonbasic_lower` may be enough because finite upper bounds are not yet supported.

### 7.5 Revised Simplex Iteration

Describe the iteration conceptually:

```text
B = basis matrix
x_B = B^{-1} b
pi^T = c_B^T B^{-1}
reduced_cost_j = c_j - pi^T A_j
choose entering column
solve B d = A_j
ratio test
pivot
```

Explain the sign convention for maximization carefully.

Do not write production code.

### 7.6 Solution Information

Explain how revised simplex should eventually populate:

```text
primal_values
slack_values
reduced_costs
basis_status
dual_values
objective_value
```

For dual values, clarify that they should be added only when the convention is well tested.

### 7.7 Reoptimization Interface

Define the future need for warm starts:

```python
solve(model, basis=None)
```

or similar.

Explain why this matters for:

```text
branch-and-bound
column generation
Benders decomposition
small RHS or bound changes
```

Do not implement the interface yet unless it is purely documented.

### 7.8 Module Layout

Recommend a future layout such as:

```text
src/silo/lp/simplex/revised.py
src/silo/lp/simplex/basis.py
src/silo/lp/simplex/factorization.py
src/silo/lp/simplex/standard_form.py
```

Explain each file’s purpose.

### 7.9 Testing Strategy

Specify tests that future implementation should pass:

```text
same LPs as tableau solver
Phase I feasibility examples
infeasible LPs
unbounded LPs
known reduced-cost examples
basis-status examples
degenerate smoke cases
comparison against tableau on small instances
```

### 7.10 Implementation Phases

Break future revised simplex implementation into small tasks:

```text
Phase 3A: standard-form builder and Basis dataclass
Phase 3B: primal revised simplex for already feasible bases
Phase 3C: Phase I basis construction
Phase 3D: solution diagnostics parity with tableau
Phase 3E: warm-start/reoptimization interface
```

### 7.11 Out of Scope

Explicitly state that the first revised simplex implementation should not include:

```text
sparse LU factorization
eta updates
dual simplex
presolve
finite upper bounds
nonzero lower bounds
MIP integration
external solver backends
performance benchmarking
```
