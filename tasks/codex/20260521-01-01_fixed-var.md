## 2. Goal

Implement conservative fixed-variable elimination in the presolve layer.

A variable is fixed when:

```text
variable.bounds.lower == variable.bounds.upper
```

within exact dataclass value comparison or existing tolerance if appropriate.

For each fixed variable:

1. remove it from the returned presolved model;
2. substitute its value into every constraint RHS;
3. remove its coefficient from every constraint;
4. add its objective contribution to the objective constant;
5. record a deterministic `ReductionRecord`;
6. record the variable name in `PresolveDiagnostics.fixed_variables`;
7. provide a recovery function or method that restores the fixed variable value in the public `Solution`.

This task should not connect presolve to CLI or default solver flow.

## 6. Mathematical Transformation

Suppose variable `x_j` is fixed:

```text
x_j = v
```

For each constraint:

```text
sum_i a_i x_i + a_j x_j sense rhs
```

substitute `x_j = v`:

```text
sum_i a_i x_i sense rhs - a_j v
```

Then remove coefficient `a_j` from the constraint.

For the objective:

```text
c^T x + c0
```

substitute:

```text
c_without_j^T x_without_j + (c0 + c_j v)
```

Then remove `x_j` from objective coefficients.

All updates should be deterministic.

Clean near-zero coefficients and RHS values using existing tolerance conventions.

## 7. Fixed Variable Detection

Use the existing `Bounds.is_fixed()` method if appropriate.

A variable with:

```text
lower == upper
```

should be eliminated.

Examples:

```text
x lower=2, upper=2 -> fixed value 2
x lower=0, upper=0 -> fixed value 0
```

Do not eliminate general variables with nonzero lower bound unless `lower == upper`.

Do not interpret binary or integer fixed variables differently in this task. If the model validates and a variable is fixed, substitution is allowed. However, be careful not to imply that general MIP solving is supported.

## 9. Fixed-Variable Reduction Records

Use:

```text
ReductionType.FIXED_VARIABLE
```

Recommended record:

```python
ReductionRecord(
    reduction_type=ReductionType.FIXED_VARIABLE,
    target=variable.name,
    description="Eliminated fixed variable.",
    data=reduction_data(
        value=fixed_value,
        objective_coefficient=objective_coefficient,
        objective_contribution=objective_coefficient * fixed_value,
    ),
)
```

If useful, include affected constraints information, but keep data small and deterministic.

Do not store large nested mutable objects in reduction data.

## 11. Solution Reconstruction

Add a conservative recovery function or method.

Preferred design:

```python
def recover_solution(
    presolve_result: PresolveResult,
    solution: Solution,
) -> Solution:
    ...
```

or:

```python
class PresolveResult:
    def recover_solution(self, solution: Solution) -> Solution:
        ...
```

Choose one clear design and test it.

### 11.1 Required recovery behavior

Given a solution for the presolved model, recovery should:

1. copy all existing `primal_values`;
2. add fixed variable values back into `primal_values`;
3. preserve or update `objective_value`;
4. preserve `status`;
5. preserve `message`;
6. preserve existing `dual_values`, `reduced_costs`, `basis_status` as much as reasonable;
7. not expose transformed-only variables.

Because fixed-variable substitution updates the objective constant, the solver-space objective should already include the fixed contribution. Recovery should not double count it.

### 11.2 Objective value

Do not recompute objective value unless necessary.

If `solution.objective_value` is not `None`, keep it as is because the presolved model's objective constant should already include fixed-variable contribution.

Tests should verify that the recovered objective equals the original objective value.

### 11.3 Slack values

For this task, there are two acceptable approaches:

Option A, simpler:

- preserve existing `slack_values`;
- because constraints keep original names and RHS is adjusted, slack values should often match original-space slacks after fixed substitution.

Option B, more robust:

- recompute slack values from the original constraints and recovered primal values if the original model is available in the reduction metadata or `PresolveResult`.

Preferred: choose the simpler approach if it keeps the design small, but document in the report whether slack recomputation is deferred.

Do not overbuild a full reconstruction framework in this task.

### 11.4 Basis status and reduced costs

Fixed variables are no longer in the presolved solver result.

For recovered `basis_status`, add fixed variables with status:

```text
fixed
```

This introduces a public status value that was previously planned for future use.

For recovered `reduced_costs`, it is acceptable to add fixed variables with reduced cost:

```text
0.0
```

or omit them. Choose one behavior and test it. Prefer adding `0.0` for fixed variables for completeness.

Do not implement dual-value recovery.
