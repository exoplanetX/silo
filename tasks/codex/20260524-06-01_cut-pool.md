## Objective

Add a deterministic cut pool with duplicate detection, activation, and scope-clearing
tests, building only on the Phase 6 cut candidate dataclasses.

## Context

Phase 6A design is recorded in `notes/18_cut_callback_boundary_design.md`.

Phase 6B added immutable `CutCandidate` and `CutMetadata` dataclasses with validation and
canonical-key behavior.

The next approved Phase 6 step is to implement the cut-pool lifecycle only. This task must
not integrate cuts into branch-and-bound and must not implement separators or callbacks.

## Scope Lock

This task solves exactly one primary problem: implement a small deterministic in-memory
cut pool for accepted cut candidates.

The implementation must:

- accept `CutCandidate` instances;
- detect duplicates using `CutCandidate.canonical_key()`;
- keep stored cuts in deterministic insertion order;
- expose accepted or active cuts for a node without mutating solver state;
- support global and node-local validity scopes;
- clear expired node-local cuts by explicit active-node scope;
- include deterministic unit tests for duplicate detection, activation, and clearing.

## Acceptance Criteria

- `CutPool` accepts unique `CutCandidate` instances and stores accepted cuts
  deterministically.
- Adding a duplicate candidate returns duplicate status without appending a second stored
  cut.
- Global cuts are active for every queried node.
- Node-local cuts are active only for their matching node.
- Clearing expired node-local cuts removes only node-local cuts whose node id is not in
  the active-node set.
- Public exports from `silo.cuts` include the cut-pool API.
- All required checks pass.
- No branch-and-bound, separator, callback, CLI, schema, or example behavior is changed.
