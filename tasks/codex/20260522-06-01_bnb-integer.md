## 1. Background

SILO now has a first native branch-and-bound solver for small binary maximization MIPs.

Current `BranchAndBoundSolver` supports:

```text
binary variables with [0, 1] bounds
optional continuous variables with lower bound 0 and no finite upper bound
maximization
linear constraints
depth-first search
first fractional binary variable branching
```

The current solver explicitly rejects general integer variables.

The MIP-to-LP relaxation builder already supports bounded nonnegative integer variables by relaxing them to continuous variables and adding explicit generated upper-bound rows:

```text
x <= U
```

This task extends branch-and-bound from binary-only to bounded nonnegative integer variables.

## 6. Supported Model Scope After This Task

After this task, `BranchAndBoundSolver` should support:

```text
maximize c^T x + c0
subject to linear rows with <=, >=, =
           binary variables x_b in {0, 1}
           bounded nonnegative integer variables x_i in {0, 1, ..., U_i}
           optional continuous variables x_c >= 0 with no finite upper bound
```

Bounds required:

### Binary variables

```text
lower = 0
upper = 1
type = binary
```

### Integer variables

```text
lower = 0
upper = finite integer U
type = integer
```

### Continuous variables

```text
lower = 0
upper = infinity
type = continuous
```

Unsupported inputs should return `SolverStatus.ERROR` from `solve()` with clear messages.

## 7. Validation Changes

Update the current binary-only validation.

Current behavior rejects `VariableType.INTEGER`.

New behavior should:

1. accept bounded nonnegative integer variables;
2. reject integer variables with infinite upper bound;
3. reject integer variables with non-integer upper bound;
4. reject integer variables with nonzero lower bound;
5. reject binary variables not exactly `[0, 1]`;
6. reject continuous variables with finite upper bounds;
7. reject minimization.

Suggested internal helper rename:

```text

```

```text
_validate_supported_mip_scope
```

It should return:

```python
tuple[str, ...]
```

containing all integer-restricted variable names in original model order, including both binary and integer variables.

Example:

```python
("x_binary", "y_integer")
```

Do not return only binary variables anymore.

## 8. Branching Behavior

Existing branching helpers already support general integer-like branching:

```text
x <= floor(value)
x >= ceil(value)
```

Keep the same rule:

```text
first fractional integer-restricted variable in model variable order
```

Integer-restricted variables include:

```text
binary variables
integer variables
```

For example, if:

```text
x = 1.7
```

then branch into:

```text
x <= 1
x >= 2
```

For a binary variable with:

```text
x = 0.4
```

branch into:

```text
x <= 0
x >= 1
```

No branching priority differences between binary and integer variables in this task.

Do not implement domain propagation beyond adding branching constraints.

## 10. Integer Feasibility

Use current integer tolerance:

```python
DEFAULT_INTEGER_TOLERANCE
```

A solution is integer-feasible if all binary and integer variables satisfy:

```text
abs(x_j - round(x_j)) <= tolerance
```

When storing an incumbent candidate, round all integer-restricted variables within tolerance.

Example:

```text
1.0000000001 -> 1.0
2.9999999999 -> 3.0
```

Do not round continuous variables.

## 11. Pruning and Status Behavior

Keep the existing pruning order:

```text
1. LP infeasible -> prune
2. LP error/numerical issue -> terminate with corresponding status
3. LP unbounded -> return UNBOUNDED
4. bound dominated -> prune
5. integer-feasible -> incumbent update and prune
6. otherwise branch
```

No change to node selection.

No change to incumbent semantics.

No change to logging semantics except messages may say "integer-feasible" instead of "binary-feasible" where appropriate.
