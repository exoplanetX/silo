# Solver Roadmap

SILO starts with Python because the first objective is not speed. The first objective is to expose the architecture of a mathematical programming solver in a form that can be inspected, tested, and revised. Python makes data structures explicit, lowers the cost of experimentation, and lets the project separate mathematical conventions from implementation tricks. A native backend can be valuable later, but only after the reference behavior is stable enough to test against.

Linear programming comes before mixed-integer programming because LP is the relaxation layer used repeatedly by MIP solvers. Branch-and-bound is not a separate world: at each node, it modifies bounds or constraints and solves an LP relaxation. If the LP layer is unclear, the MIP layer will become a collection of opaque control-flow decisions. SILO therefore treats LP status, primal solution, dual values, reduced costs, and basis information as first-order concepts before building tree search.

Tableau simplex comes before revised simplex for pedagogical reasons. A tableau implementation is not the right path to high performance, but it displays the algebra of pivoting, feasibility, reduced costs, and unboundedness in one object. It is suitable for small deterministic tests and for learning the connection between canonical form and simplex iterations. Revised simplex should follow once these conventions are fixed, because basis representation, factorization, and reoptimization are easier to understand when the tableau logic is already tested.

MIP is treated as an upper layer over LP relaxation. This means branch-and-bound should not duplicate LP logic or smuggle external solver behavior into native algorithm modules. A node should represent local bounds and branching decisions. The LP solver should return a status and relaxation solution. The MIP layer should update incumbent, global bound, and open-node state from that information. This separation will make it possible to compare depth-first search, best-bound search, branching rules, and later cuts without rewriting the LP layer.

Stochastic and robust optimization enter SILO first as model transformations. In early phases, uncertainty should not be presented as a new solver. A scenario model can be expanded into a deterministic equivalent; a simple robust counterpart can be constructed from an uncertainty set when the mathematical assumptions permit it. This keeps uncertainty modules close to modeling and canonicalization, while LP, MIP, and decomposition solvers continue to operate on ordinary deterministic model objects.

The project should remain intentionally modest in the early phases. It should not claim industrial numerical behavior, sparse factorization quality, or competitive performance. The reference value comes from clean interfaces, visible conventions, small known-solution instances, and deterministic tests. Once those are reliable, the same architecture can support revised simplex, presolve, branch-and-bound, cut pools, decomposition, and native kernels.
